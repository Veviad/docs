Veviad Central is the internal control plane for customer records, application instances, Trial automation, infrastructure configuration, and operational support.

<Warning>
  All signed-in users can see the main navigation. Directus permissions still decide whether an operation succeeds. Treat the responsibilities below as an operating model, not as a technically enforced role matrix.
</Warning>

## Intended responsibilities

| Responsibility | Typical work |
| --- | --- |
| Support | Review customers, instances, feedback, notifications, and audit history. Escalate infrastructure or destructive actions. |
| Operations | Provision and operate instances, run backups and restores, manage updates, and handle Trial retries. |
| Platform administration | Configure servers, deployment templates, DNS, SMTP, backup storage, API credentials, and global policies. |

## Start with the dashboard

The dashboard summarizes customer, instance, server, and published-template counts. It also shows server capacity, instance health distribution, and recently created instances. Use it to spot capacity pressure or a growing unhealthy population, then open the relevant record for detail.

## Status principles

- Long-running work is asynchronous. Transitional states such as `provisioning`, `upgrading`, `restoring`, `pausing`, or `decommissioning` are expected.
- Do not repair a stuck workflow by editing status fields directly in Directus. Check notifications, instance activity, the audit log, and the relevant retry action first.
- A green UI message confirms that an action was accepted. Confirm the final state and health separately.
- Recovery and decommission workflows enforce a verified backup. Database reset, downgrade, and updates with **Require pre-upgrade backup** disabled may not; manually confirm a current backup before any destructive change.

## Before any high-impact action

1. Confirm the customer and exact instance, including domain and server.
2. Read the current status, health result, and recent activity.
3. Confirm a current verified full backup when data may be removed or replaced.
4. State the expected downtime and final state.
5. After the action, verify status, health, DNS, and the audit trail.
