People (/people) is the workforce directory. A person can receive assets, licenses, accessories, and organizational assignments without having an application account.

Application access is a separate entitlement. Creating or importing a person does not assign a role, create credentials, send an invitation, or allow sign-in.

## People and application users

| Record | Purpose | Can sign in? |
|---|---|---|
| Person | Workforce identity used for assignments and history | Only when application access is active |
| Application user | A person with pending, active, or suspended application access | Depends on person status and access status |

Use **People** for workforce records. Use **Admin → Users** for login access, authentication methods, roles, and invitations.

![The People directory with workforce records and assignment-ready organization data](/images/screenshots/people.webp)

## Directory views

The People submenu provides these standard views:

- **All People** — every person visible to you
- **Active** — people available for new assignments
- **Inactive** — retained workforce records that are no longer assignable
- **Managers** — people configured as workforce managers

The name in the **Person** column opens the read-only profile. Depending on your permissions, row actions also let you view the profile or open the administrative editor.

## Add and import people

Choose **Add Person** to create one workforce record, or **Import People** for a CSV-based directory import.

A new person starts with:

- person status **Active**;
- application access **None**;
- no role;
- no password or SSO entitlement;
- no invitation email.

This makes it practical to import a complete workforce directory even when only a small subset uses Veviad.

## View and edit a person

The profile at /people/{id} is a read-only directory view.

Administrators and delegated roles with the user.edit permission can open **Admin → People** at /admin/people/{id}. The administrative editor contains profile and organization fields plus linked assets, licenses, history, and relationship analysis.

When the person has no application access, the editor shows **No application access** and a **Grant Access** action. The Roles tab is unavailable because roles only control authenticated application capabilities.

![The administrative person editor for a workforce record without application access](/images/screenshots/admin-person-no-access.webp)

## Person lifecycle

Authorized administrators can change the person's Active switch in the administrative editor.

Deactivating a person:

- preserves existing assignments, custody records, and audit history;
- removes the person from new assignment pickers;
- keeps the independent application-access entitlement unchanged;
- immediately blocks effective sign-in and revokes active sessions and API tokens when access exists.

Reactivating restores workforce status only. It does not grant access or change a role.

For directory-SCIM-managed people, profile and lifecycle fields are read-only in Veviad. Make those changes in the identity provider.

## When to use each area

| Task | Area |
|---|---|
| Add or import workforce records | People |
| View assignments and workforce history | People |
| Edit profile or organization data | Admin → People |
| Activate or deactivate a workforce record | Admin → People |
| Grant, suspend, or restore application access | Admin → Users or the administrative person editor |
| Send or manage a local invitation | Admin → Users / Invitations |
| Assign an application role | Admin → People, after access exists |
| Manage an identity-provider-owned record | Identity provider |
