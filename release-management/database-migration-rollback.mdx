An application rollback is safe only when the database schema remains compatible with the older release.

## Decision order

1. Identify forward- and backward-incompatible migrations.
2. Determine whether code can roll back without a schema change.
3. For additive migrations, prefer a code rollback with the newer schema when verified compatible.
4. Use <code>migrate:rollback</code> only after testing the exact migration's reverse path.
5. Restore a backup when data was transformed or removed and no safe down path exists.

## Evidence before change

Record backup time, database version, release SHA, migration list, and expected downtime. Verify restore access and destination.

## Verification

After rollback or restore, check migration status, representative row counts, sign-in, key assets, search, queues, and <code>/health</code>.

<Warning>
  Never run a blanket rollback command against an unknown production migration set.
</Warning>
