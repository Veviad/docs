This handbook is the maintainer guide for Veviad production releases. It defines where changes merge, how stable maintenance branches work, how Docker images are validated, and how to cut stable, prerelease, and patch releases without pulling future features into a security fix.

The goal is simple:

- `develop` is the rolling integration line for the next planned release.
- `release/X.Y` is the stable maintenance line for supported `X.Y.z` patches.
- GitHub Releases and Docker tags are the production source of truth.
- Production never tracks a moving tag such as `develop`, `latest`, or `dev-*`.

## Best-practice basis

This policy combines three practices that fit Veviad's current release pipeline:

- [Semantic Versioning](https://semver.org/): `MAJOR.MINOR.PATCH`, where PATCH is for backward-compatible fixes, MINOR for backward-compatible features, and MAJOR for incompatible changes.
- Release branches: keep a branch for a supported stable line when patches must ship while `develop` continues moving.
- [GitHub Releases and release assets](https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases): publish exact version tags, image digest, source commit, SBOMs, and attestation status with every production release.

## Branch model

| Branch or tag | Purpose | Merges from | Merges to |
| --- | --- | --- | --- |
| `develop` | Rolling integration branch for the next planned release | `feature/*`, normal `fix/*`, forward-ported hotfixes | Future `release/X.Y` branches |
| `feature/<issue>-<name>` | Temporary branch for planned feature work | `develop` | `develop` |
| `fix/<issue>-<name>` | Temporary branch for non-emergency fixes for the next release | `develop` | `develop` |
| `release/X.Y` | Stable maintenance branch for production line `X.Y.z` | selected cherry-picks only | production tags `vX.Y.z` |
| `hotfix/vX.Y.Z-<name>` | Temporary urgent fix branch for a production patch | `release/X.Y` | `release/X.Y`, then forward-port to `develop` |
| `vX.Y.Z` | Immutable Git tag for a production release | chosen release commit | never moves |

Rules:

- Feature work normally starts from `develop` and merges back to `develop`.
- Patch releases start from the supported maintenance branch, not from `develop`, once `develop` contains future work.
- Never merge `develop` into `release/X.Y`.
- Bring selected fixes into `release/X.Y` with cherry-pick or a hotfix PR.
- Always forward-port production fixes to `develop` so the next release does not regress the patch.

## Version policy

| Version type | Example | Source branch | Meaning |
| --- | --- | --- | --- |
| Stable first release | `v1.0.0` | `develop` at first release cut | First production release |
| Stable minor release | `v1.1.0` | `release/1.1` | Planned feature release |
| Stable patch release | `v1.0.1` | `release/1.0` | Production or security fix only |
| Release candidate | `v1.1.0-rc.1` | `release/1.1` | Candidate for planned stable release |
| Development build | `1.2.0-dev.YYYYMMDD.N` | `develop` | Rolling dev image only |

Patch releases must stay in the same major/minor line and increment PATCH by one:

- `v1.0.0` to `v1.0.1` is valid.
- `v1.0.0` to `v1.0.2` is invalid if `v1.0.1` does not exist.
- `v1.0.0` to `v1.1.0` is a minor release, not a patch.

After a stable minor release, keep `BASE_VERSION` on `develop` ahead of the latest stable line. Example: after `v1.1.0`, `develop` should normally move to `1.2.0-dev...`.

## Automation status

Current tracked workflow facts:

- `.github/workflows/ci.yml` runs for `develop` pushes and pull requests.
- `.github/workflows/deploy-develop.yml` builds rolling development images from `develop` after CI succeeds.
- `.github/workflows/release-image.yml` publishes production images from a GitHub Release after it verifies CI success, a `sha-<short>` candidate image, tag validity, image metadata, SBOMs, and release evidence.

Professional maintenance branches require the same CI and candidate-image guarantees as `develop`. Before cutting patch releases from `release/X.Y`, ensure the workflows have been extended so:

- CI runs on `release/**` pushes and pull requests.
- A candidate image is built for `release/**` commits and tagged with `sha-<short>`.
- The production release workflow accepts `vX.Y.Z` only from the matching `release/X.Y` branch when `PATCH > 0`.

Until those workflow changes exist, do not claim a `release/X.Y` patch is ready for production. The branch policy is correct, but the automation must guard it.

## Normal feature release

Use this flow for planned releases such as `v1.1.0`.

1. Merge planned feature and fix PRs into `develop`.
2. Keep unfinished future work out of `develop`, or hide it behind disabled feature flags.
3. When the release scope is frozen, create the release branch:

```bash
git fetch origin
git checkout develop
git pull origin develop
git checkout -b release/1.1
git push origin release/1.1
```

4. Only stabilization fixes go into `release/1.1`.
5. Cherry-pick any stabilization fixes back to `develop` if they were made on the release branch first.
6. Validate the exact `sha-<short>` candidate image from `release/1.1`.
7. Publish `v1.1.0-rc.N` releases as prereleases if you need customer or staging validation.
8. Publish `v1.1.0` when the manual gates pass.
9. After `v1.1.0`, advance `BASE_VERSION` on `develop` to the next development line, normally `1.2.0`.

## Patch or security release

Use this flow for urgent production fixes such as `v1.0.1` while `develop` already contains `1.1.0` work.

1. Confirm the supported stable line exists:

```bash
git fetch origin --tags
git checkout release/1.0
git pull origin release/1.0
```

2. Create a hotfix branch from the maintenance branch:

```bash
git checkout -b hotfix/v1.0.1-security-fix release/1.0
```

3. Apply the smallest safe fix and add regression tests.
4. Open a PR into `release/1.0`.
5. Require CI, code review, Codacy, and security review as appropriate.
6. Merge into `release/1.0`.
7. Confirm a `sha-<short>` candidate image exists for the merged hotfix commit.
8. Publish `v1.0.1` from that exact commit.
9. Forward-port the same fix into `develop` immediately:

```bash
git checkout develop
git pull origin develop
git cherry-pick -x <hotfix_commit_sha>
git push origin develop
```

10. If the cherry-pick conflicts, resolve it on `develop` and keep the same behavior covered by tests.

Do not merge unrelated features from `develop` into `release/1.0`. A patch release should contain the previous stable release plus the selected fix.

## Manual release gates

Complete these gates before publishing any stable GitHub Release.

| Gate | Required evidence |
| --- | --- |
| Exact candidate image validated | `sha-<short>` tag, digest, staging URL or test notes |
| CI green | Successful CI run for the release source commit |
| Release notes reviewed | Non-empty notes with changes, fixes, and operator impact |
| Migration impact assessed | Migration list and backward-compatibility classification |
| Backup and rollback plan confirmed | Prior release tag, digest, backup/restore plan |
| Security review complete when applicable | Advisory, CVE, scanner result, or reviewer sign-off |

Capture the source commit once and use it throughout the release:

```bash
RELEASE_SHA=<full-40-character-commit-sha>
SHORT_SHA=$(printf '%s' "$RELEASE_SHA" | cut -c1-7)
IMAGE=ghcr.io/veviad/veviad_app
SHA_TAG="$IMAGE:sha-$SHORT_SHA"

docker buildx imagetools inspect "$SHA_TAG"
VALIDATED_DIGEST=$(docker buildx imagetools inspect "$SHA_TAG" \
  --format '{{json .Manifest.Digest}}' | tr -d '"')
echo "Validated digest: $VALIDATED_DIGEST"
```

The production workflow will resolve and verify the candidate image itself. If the candidate image is missing, the release should fail before publishing.

## Release notes checklist

Stable release notes must include:

- Version and release date.
- Source branch and source commit.
- User-visible changes.
- Security fixes, with sensitivity appropriate to disclosure timing.
- Database migrations and whether each is backward compatible.
- Upgrade steps or operator action.
- Rollback guidance.

Do not publish empty or placeholder notes.

## Cut a GitHub Release

Create the annotated tag from the validated commit:

```bash
RELEASE_TAG=v1.1.0
git tag -a "$RELEASE_TAG" "$RELEASE_SHA" -m "Release $RELEASE_TAG"
git push origin "$RELEASE_TAG"
```

Create the release:

```bash
gh release create "$RELEASE_TAG" \
  --target "$RELEASE_SHA" \
  --title "$RELEASE_TAG" \
  --notes-file RELEASE_NOTES.md \
  --verify-tag
```

For a release candidate:

```bash
RELEASE_TAG=v1.1.0-rc.1
gh release create "$RELEASE_TAG" \
  --target "$RELEASE_SHA" \
  --title "$RELEASE_TAG" \
  --notes-file RELEASE_NOTES.md \
  --verify-tag \
  --prerelease
```

Publishing the release triggers `.github/workflows/release-image.yml`.

## Monitor the production workflow

Watch the release workflow:

```bash
gh run watch --workflow release-image.yml
```

The workflow should:

1. Validate the tag grammar and prerelease flag.
2. Verify CI success for the source commit.
3. Verify a `sha-<short>` candidate image exists.
4. Enforce the PATCH\+1 rule for patch releases.
5. Resolve the tested candidate image digest and embedded source metadata.
6. Build and publish the production image with stable production metadata.
7. Verify both `vX.Y.Z` and `X.Y.Z` tags point to the same release digest.
8. Generate SBOMs and record attestation availability.
9. Attach `release-image.json`, SPDX SBOM, and CycloneDX SBOM.
10. Update release notes with digest and pull/verify instructions.

The current workflow validates the prebuilt candidate image, then builds the production image with stable metadata so the app reports `X.Y.Z` instead of a development version. The release evidence records the production image digest that customers should deploy.

## Post-release verification

Verify both release tags:

```bash
IMAGE=ghcr.io/veviad/veviad_app
RELEASE_TAG=v1.1.0
CORE_VERSION=${RELEASE_TAG#v}

docker buildx imagetools inspect "$IMAGE:$RELEASE_TAG"
docker buildx imagetools inspect "$IMAGE:$CORE_VERSION"

DIGEST_V=$(docker buildx imagetools inspect "$IMAGE:$RELEASE_TAG" \
  --format '{{json .Manifest.Digest}}' | tr -d '"')
DIGEST_CORE=$(docker buildx imagetools inspect "$IMAGE:$CORE_VERSION" \
  --format '{{json .Manifest.Digest}}' | tr -d '"')

test "$DIGEST_V" = "$DIGEST_CORE"
echo "Release digest: $DIGEST_V"
```

Verify the release assets:

```bash
gh release view "$RELEASE_TAG" --json assets,body
gh release download "$RELEASE_TAG" \
  --pattern release-image.json \
  --pattern sbom.spdx.json \
  --pattern sbom.cyclonedx.json
```

If attestation is available:

```bash
gh attestation verify "oci://$IMAGE@$DIGEST_V" \
  --repo veviad/veviad_app
```

If attestation persistence was unavailable, confirm `release-image.json` records the unavailable state and do not describe the release as fully attested.

## Branch protection and rulesets

Configure GitHub rulesets or branch protections for:

- `develop`
- `release/**`
- `v*` tags

Recommended protections:

- Require pull requests.
- Require CI status checks.
- Require conversation resolution.
- Require linear history if the team wants cleaner maintenance cherry-picks.
- Block force-pushes.
- Block branch deletion for active `release/**` branches.
- Restrict release tag creation and block tag movement or deletion.

Do not enable immutable GitHub Releases until the release workflow can attach all assets before publication. The current workflow updates release notes and uploads evidence after the release is published; immutable releases would require a draft-first release flow.

## Failure handling

| Failure | Meaning | Action |
| --- | --- | --- |
| Invalid tag | Tag is not `vX.Y.Z` or `vX.Y.Z-rc.N` | Delete the bad release/tag and recreate correctly |
| Prerelease mismatch | `-rc.N` tag not marked prerelease, or stable tag marked prerelease | Fix release metadata and republish |
| CI gate failed | Source commit is not green | Fix and rerun CI before releasing |
| Missing `sha-<short>` image | Candidate image was not built | Run or fix candidate image workflow before releasing |
| Invalid PATCH\+1 | Patch skips or changes major/minor | Use the next PATCH only |
| Immutability conflict | Tag already points to different contents | Stop; never overwrite a released version |
| SBOM failure | Evidence incomplete | Fix workflow/tooling and rerun |
| Attestation unavailable | GitHub could not persist attestation | Release can continue only if evidence records the unavailable state |

## Decommissioning release branches

Keep `release/X.Y` only while customers may need patches on that line. When a line is no longer supported:

1. Confirm no supported customer instance remains on `X.Y.z`.
2. Confirm final release tags and release evidence exist.
3. Announce end of maintenance for that line.
4. Delete the branch after support ends:

```bash
git push origin --delete release/X.Y
```

Never delete release tags. Tags and release evidence remain the audit trail.
