The Admin area separates workforce management from application access. Its submenu contains **General**, **People**, **Users**, **Invitations**, **Roles**, **Integrations**, and **Login Links**.

## People

**Admin → People** opens the workforce directory. It includes everyone who may receive assignments, including people who cannot sign in.

Selecting Edit opens the canonical administrative editor at /admin/people/{id}. This editor is available for people with or without application access.

For every person, authorized administrators can manage:

- profile and organization details;
- person status;
- linked assets and licenses;
- lifecycle history and relationship analysis.

Application-account controls appear only after access exists.

## Users

**Admin → Users** lists only people whose application access is **Pending**, **Active**, or **Suspended**. Workforce-only records with access **None** remain in People and can be promoted with **Grant Access**.

The user directory shows application-access status, authentication method, role, and management authority. The name opens the read-only profile; Edit opens the same administrative person editor used from People.

## Grant application access

Choose **Grant Access** and select an existing active person without access.

1. Select the person.
2. Select the sign-in method.
3. Confirm or add the email address.
4. Select the application role.
5. Grant access.

For a local account, Veviad sends an invitation and sets access to **Pending** until the recipient establishes credentials. For SSO, access can become active without a local password.

Roles are assigned at this stage because they answer what an authenticated person may do. A workforce-only person has no role.

If Access SCIM owns the person's entitlement, access and role changes are read-only in Veviad. Manage them in the identity provider.

## Independent statuses

| Status | Values | Meaning |
|---|---|---|
| Person status | Active, Inactive | Whether the workforce record can receive new assignments |
| Access status | None, Pending, Active, Suspended | Whether an application account exists and its lifecycle |
| Effective login | Eligible or blocked | Calculated from both statuses and the authentication configuration |

Authentication requires both person status **Active** and access status **Active**.

Deactivating a person does not erase access history or roles. Suspending access does not deactivate the workforce record.

## Invitations

Local invitations reuse an existing person when possible and do not create a duplicate workforce record.

- **Pending** — invitation is valid but credentials are not established
- **Accepted** — access is active
- **Expired** — generate a replacement when access is still intended
- **Revoked** — the invitation can no longer be used

An active pending invitation for the same person or normalized email cannot be created twice.

## Roles

Roles only apply to application users. The Roles tab is disabled when access is **None**.

System and custom roles define authenticated permissions. Role assignment requires user.assign.role. Access SCIM group mappings can manage roles from the identity provider; People Directory SCIM groups never assign roles.

## Permissions and ownership

Administrative actions use permissions rather than a hard-coded role:

- user.view controls directory visibility;
- user.edit controls the administrative editor;
- user.create controls adding people and granting access;
- user.delete controls person lifecycle and access suspension;
- user.assign.role controls role changes.

The default Admin role receives these permissions. Custom delegated HR or IT roles can be created without changing the underlying workflow.

For SSO and the two SCIM provisioning purposes, see [SSO and SCIM](/organization/sso-scim).
