Veviad separates **profile data**, **person status**, **application access**, **authentication**, and **roles**. Choose an owner for each concern before enabling SCIM, SSO, or JIT.

<Info>
  **Entra groups are the only external role source.** Veviad does not use Entra app roles, SCIM User `roles`, the `roles` claim, or directory-role `wids` for authorization.
</Info>

## Choose a flow

| Flow | Profile and person status | Application access | Authentication | Roles |
| --- | --- | --- | --- | --- |
| Local person and invitation | Local | Local invitation | Local after activation | Local |
| SCIM People Directory only | Workforce SCIM | None | None | None |
| SCIM People Directory plus local access | Workforce SCIM | Local | Local password or permitted SSO | Local |
| Access SCIM, SSO off | Local or Workforce SCIM | Access SCIM | **Local — activation pending**, then **Local** after invitation acceptance | Access SCIM groups |
| Access SCIM, SSO on | Local or Workforce SCIM | Access SCIM | SSO | Access SCIM groups |
| Both SCIM feeds | Workforce SCIM | Access SCIM | Local activation or SSO | Access SCIM groups |
| SSO for existing users | Local or Workforce SCIM | Already present | SSO | Local when sync is off; SSO groups when sync is on |
| SSO with controlled JIT | Provider data at first login | Created or promoted at login | SSO | SSO groups; synchronization is mandatory |
| Local exception while Access SCIM is active | Local | Local invitation or grant | Local password or permitted SSO | Local |

<Note>
  A local exception remains outside Access SCIM scope. Removing the person from the IdP does not revoke that local access; manage it explicitly in Veviad.
</Note>

## Authority precedence

| Concern | Effective authority |
| --- | --- |
| Profile and person status | SCIM People Directory, when the person is managed by that feed |
| Application entitlement | Access SCIM for SCIM-managed access; otherwise local administration |
| External roles | Access SCIM groups first. SSO groups are authoritative only while Access SCIM is disabled |
| Local roles | Used only when neither Access SCIM groups nor effective SSO group synchronization controls roles |
| Authentication | Local password and SSO can coexist independently of role authority |

## Recommended baseline

- Keep the two SCIM purposes in separate enterprise applications with separate bearer tokens and scoped populations.
- Use immutable Entra group object IDs and direct membership.
- Set the default role to **None** unless a deliberate fallback is required.
- Pilot one representative user from every mapped group before full provisioning.
- Keep a tested local administrator recovery path unless **Force SSO** is an intentional policy.
