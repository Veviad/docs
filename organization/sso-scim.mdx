Veviad can integrate with Microsoft Entra ID for two related identity workflows:

- **Single Sign-On (SSO):** users authenticate with Microsoft Entra ID.
- **SCIM provisioning:** Microsoft Entra ID creates, updates, and deactivates Veviad users automatically.

Role mapping connects those workflows to Veviad permissions. When Entra sends group memberships that match a configured role mapping, Veviad can assign the corresponding local role, such as `admin`, `asset-manager`, `department-manager`, or `user`.

## Before you start

You need:

- administrator access in Veviad with permission to edit settings;
- administrator access in Microsoft Entra ID for app registration and enterprise application configuration;
- the public Veviad application URL, for example `https://app.example.com`;
- the Microsoft Entra tenant ID;
- the Microsoft Entra application client ID and client secret;
- the Entra group Object IDs that should map to Veviad roles.

Use Entra group **Object IDs** for role mappings whenever possible. Display names can change; Object IDs are stable and are the default value Microsoft Entra emits in group claims.

## How Veviad uses groups

Veviad stores one identity-provider group value per local role. The value can be an Entra group Object ID or a group name, but it must match the value Entra sends to Veviad exactly.

| Veviad area | Trigger | Group source | What happens |
|---|---|---|---|
| SSO login | User signs in through Microsoft Entra ID | OIDC token `groups` claim | Veviad can sync roles during login when **Sync Roles on Login** is enabled. |
| SCIM provisioning | Microsoft Entra creates or updates a user | SCIM `groups` attribute | Veviad considers the SCIM group memberships and applies matching role mappings. |

SSO and SCIM are independent triggers. A user can be provisioned by SCIM before they ever sign in, and a user's roles can also be refreshed during a later SSO login.

## Configure SSO in Veviad

![The SSO configuration modal with the SSO / OIDC tab open](/images/screenshots/sso-oidc.webp)

_View shown: `demo.veviad.cloud/admin/integrations`_

Open `Organization → Admin → Integrations`, then edit **Single-Sign-On**.

On the **SSO / OIDC** tab:

1. Enable **Enable SSO**.
2. Set **Provider** to `Azure Entra ID`.
3. Select the appropriate **Cloud Instance**.
4. Enter a **Login Button Label** that users will recognize, such as `Microsoft Azure`.
5. Decide whether to enable **Force SSO**.
6. Copy the **Redirect URI / Reply URL** shown by Veviad.
7. Enter the Microsoft Entra **Tenant ID**.
8. Enter the Microsoft Entra application **Client ID**.
9. Enter the Microsoft Entra application **Client Secret**.
10. Use **Test Connection**.
11. Save the configuration.

When **Force SSO** is enabled, the standard email and password form is hidden for normal users. Keep a documented break-glass administrator account and confirm the administrator bypass works before enforcing SSO for the whole tenant.

## Configure Microsoft Entra ID for SSO

In Microsoft Entra ID:

1. Create or open the application registration used for Veviad.
2. Add the Veviad **Redirect URI / Reply URL** as a Web redirect URI.
3. Create a client secret and copy it into Veviad.
4. Assign the users or groups that should access Veviad to the enterprise application.
5. Configure group claims so Entra sends group membership information to Veviad.

Recommended group-claim configuration:

- Emit groups in the OIDC token used by the application.
- Prefer `ApplicationGroup` when possible so only groups assigned to the application are emitted.
- Use the default group Object ID value and enter the same Object ID in Veviad role mappings.

Microsoft Entra limits how many groups can be emitted in tokens. If a user belongs to too many groups, Entra may omit the group claim and send an overage indicator instead. Keeping the application scope tight, and using application-assigned groups, reduces that risk.

## Configure role mappings

![The Role Mapping tab with a group input per Veviad role](/images/screenshots/sso-role-mapping.webp)

_View shown: `demo.veviad.cloud/admin/integrations`_

Open the **Role Mapping** tab in the SSO configuration modal.

For each Veviad role that should be controlled by Entra:

1. Enter the Entra group Object ID in the role's input.
2. Leave roles blank when they should not be managed by Entra.
3. Select **Save Role Mappings**.
4. Close and reopen the modal to confirm the saved values are still present.

Example:

| Veviad role | Entra group value |
|---|---|
| `admin` | Object ID of the Entra administrators group |
| `asset-manager` | Object ID of the Entra asset managers group |
| `department-manager` | Object ID of the Entra department managers group |
| `user` | Object ID of the baseline access group |

Unmapped Entra groups do not block login or provisioning. They are ignored for role assignment.

## Role Sync Settings

The **Role Sync Settings** section controls fallback and SSO login synchronization.

| Setting | Effect |
|---|---|
| **Default Role** | Assigned on SSO login when no group mapping matches, and on SCIM user creation when no group mapping matches. Use this for a safe baseline role such as `user`. |
| **Sync Roles on Login** | When enabled, Veviad re-evaluates Entra group memberships every time the user signs in with SSO. |

Select **Save Sync Settings** after changing either setting.

## SSO login behavior

When a user signs in with SSO, Veviad finds or creates the user by Entra identity and email address.

If **Sync Roles on Login** is disabled:

- Veviad does not change the user's roles from Entra groups during login.
- Existing manual role assignments remain unchanged.

If **Sync Roles on Login** is enabled:

- Veviad reads group values from the Entra token.
- Each matching group mapping resolves to a Veviad role.
- Users in multiple mapped groups receive the union of all matching roles.
- Roles that are not managed by Entra mappings are preserved.
- If no group mapping matches, Veviad assigns the configured **Default Role** when one is set.
- If no group mapping matches and no default role is set, existing roles are left unchanged.

Role sync happens at login time. If a user's Entra group membership changes, the user must sign in again for SSO login sync to run.

## Configure SCIM provisioning

![The SCIM Provisioning tab with endpoint URL and bearer token](/images/screenshots/sso-scim.webp)

_View shown: `demo.veviad.cloud/admin/integrations`_

SCIM provisioning is configured after SSO is enabled.

In Veviad, open `Organization → Admin → Integrations`, edit **Single-Sign-On**, then open **SCIM Provisioning**:

1. Copy the **SCIM Endpoint URL**. It ends with `/scim/v2`.
2. Generate a **SCIM Bearer Token**.
3. Copy the token immediately. It is shown only once.

In Microsoft Entra ID:

1. Open the Veviad enterprise application.
2. Go to **Provisioning**.
3. Set provisioning mode to **Automatic**.
4. Enter the Veviad SCIM Endpoint URL as the **Tenant URL**.
5. Enter the Veviad SCIM Bearer Token as the **Secret Token**.
6. Run **Test Connection**.
7. Review user and group attribute mappings.
8. Assign the users or groups that should be provisioned.
9. Start provisioning.

If provisioning scope is set to sync only assigned users and groups, no users are provisioned until users or groups are assigned to the enterprise application.

## SCIM role behavior

When Microsoft Entra provisions or updates a user through SCIM, Veviad looks for group memberships in the SCIM payload.

On user creation:

- matching SCIM groups assign the corresponding Veviad roles;
- users in multiple mapped groups receive all matching roles;
- if no mapped group matches and a **Default Role** is configured, Veviad assigns the default role.

On user update:

- when the update contains group memberships that match role mappings, Veviad syncs the mapped roles from those groups;
- manually assigned roles that are not controlled by Entra mappings are preserved;
- if the update contains no matching group membership, existing roles are left unchanged.

SCIM provisioning does not require **Sync Roles on Login**. That toggle only controls role synchronization during SSO login.

## Operational guidance

Use one of these patterns:

| Pattern | Recommended when | Required configuration |
|---|---|---|
| SSO only | Users are created on first login and roles should update at login. | SSO enabled, group claims configured, role mappings saved, **Sync Roles on Login** enabled. |
| SCIM plus SSO | Users should exist before first login and lifecycle should follow Entra assignments. | SSO enabled, SCIM token configured, provisioning running, role mappings saved. Enable **Sync Roles on Login** if login should also refresh roles. |
| SCIM only for lifecycle, manual roles | Identity lifecycle is automated but Veviad roles are managed manually. | SCIM configured, role mappings left blank, **Sync Roles on Login** disabled. |

For most production rollouts, use **SCIM plus SSO**. SCIM handles user lifecycle, while SSO provides authentication. Enabling login role sync is useful when you want group changes to take effect on the next login as well as during provisioning cycles.

## Troubleshooting

### The user can sign in but did not receive the expected role

Check:

- the role mapping is saved in Veviad and still appears after reopening the modal;
- **Sync Roles on Login** is enabled if you expect SSO login to update roles;
- the Entra token includes a `groups` claim;
- the value in the token exactly matches the value entered in Veviad;
- the user is assigned to the enterprise application;
- the user is a member of the mapped Entra group.

### SCIM created the user but no role was assigned

Check:

- the SCIM provisioning payload includes `groups`;
- group provisioning or group membership synchronization is enabled in Entra provisioning mappings;
- the SCIM group value matches the Veviad role mapping;
- a **Default Role** is configured if users without mapped groups should receive baseline access;
- the Entra provisioning logs show the user and group membership update as successful.

### A group membership changed in Entra but Veviad still shows the old role

The update depends on the workflow:

- SSO role sync runs the next time the user signs in.
- SCIM role sync runs when Entra sends a provisioning update. You can use on-demand provisioning in Entra to test immediately.

If the user no longer has any mapped groups, Veviad preserves existing roles unless a default-role fallback or another provisioning action changes them. Review the user manually when removing the final mapped group from an account.

### The SCIM token was lost or exposed

Regenerate the token in Veviad and update the **Secret Token** in Microsoft Entra ID. Regenerating invalidates the previous token.
