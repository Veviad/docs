# Startup tasks and seeding

Container startup must be repeatable. Veviad handles this by separating normal migrations from one-time operational tasks. The task ledger lives in the application database, so multiple containers can start together without running the same task more than once.

## Startup sequence

The container startup process performs these steps:

1. Wait for the database connection.
2. Run schema migrations when migration auto-run is enabled.
3. Add required bootstrap tasks when initial admin credentials are configured.
4. Run the startup task runner when `STARTUP_TASKS` is not empty.
5. Create the initial user when `INITIAL_ADMIN_EMAIL` and `INITIAL_ADMIN_PASSWORD` are configured.
6. Continue to the container command.

Migrations are allowed to run on every deploy. Seeders, search setup, search reindexing, and database resets must go through startup tasks.

## Configuration

| Variable | Default | Purpose |
|---|---:|---|
| `STARTUP_TASKS` | empty | Comma-separated one-time tasks in `task-name:key` format. |
| `STARTUP_TASK_TIMEOUT_MINUTES` | `120` | Time before a running task is considered stale and reclaimable. |
| `STARTUP_TASK_POLL_SECONDS` | `5` | Wait interval when another container owns a task. |
| `ALLOW_DESTRUCTIVE_STARTUP_TASKS` | `false` | Required before `fresh-db-seed` can wipe and reseed the database. |

Example:

```txt
STARTUP_TASKS=essential-seeders:reference-data-v1,search-ensure-settings:search-settings-v1
```

## Task keys

A task key is the durable identity of one operation. Completed keys are skipped forever. To run the same type of operation again, use a new key.

Good keys describe why the task exists:

```txt
essential-seeders:reference-data-v1
search-ensure-settings:search-schema-v2
search-reindex:search-upgrade-YYYY-MM-DD
fresh-db-seed:test-reset-YYYY-MM-DD
```

Do not use random keys for routine deploys. A changed key means "run this task again".

## Supported tasks

| Task | What it does | Safe for production |
|---|---|---|
| `essential-seeders:<key>` | Runs reference-data seeders such as roles, categories, statuses, locations, and report templates. | Yes |
| `search-ensure-settings:<key>` | Creates or configures search indexes and waits for settings tasks in strict mode. | Yes |
| `search-reindex:<key>` | Rebuilds search documents from primary application data and waits for search tasks in strict mode. | Yes, schedule during a maintenance window for large datasets |
| `fresh-db-seed:<key>` | Resets and reseeds the database. | No, test/disposable environments only |

Fresh database resets are destructive. Use `fresh-db-seed:<key>` only in disposable environments and only when `ALLOW_DESTRUCTIVE_STARTUP_TASKS=true`.

## Concurrency behavior

When several containers start at the same time:

- one container claims a pending, failed, or stale task;
- other containers wait for completion;
- completed tasks are skipped;
- failed tasks are retried on later startups;
- stale running tasks are reclaimed after `STARTUP_TASK_TIMEOUT_MINUTES`.

This makes startup tasks suitable for horizontally scaled SaaS deployments.

## Fresh installation

For a new environment, use migrations plus safe startup tasks:

```txt
STARTUP_TASKS=essential-seeders:reference-data-v1,search-ensure-settings:search-settings-v1
ALLOW_DESTRUCTIVE_STARTUP_TASKS=false
```

If an initial admin account should be created automatically, configure `INITIAL_ADMIN_EMAIL` and `INITIAL_ADMIN_PASSWORD`. The user creation command is idempotent; it will not recreate an existing user on every restart.

## Test reset

For local or disposable test environments only:

```txt
ALLOW_DESTRUCTIVE_STARTUP_TASKS=true
STARTUP_TASKS=fresh-db-seed:test-reset-YYYY-MM-DD
```

After the reset succeeds, turn destructive startup tasks off again or change the environment back to a normal startup task set.

## Troubleshooting

Check the startup task ledger first. The most useful fields are the task key, status, attempt count, start/completion timestamps, and latest error message.

Common responses:

- `completed`: the key already ran; use a new key only when you intentionally want to rerun work.
- `failed`: fix the underlying service or data problem, then restart the container or rerun the startup task runner.
- `running`: another container may own the task. Check logs before waiting for the stale timeout.
- `last_error`: contains the most recent failure message captured by the runner.

Manual runs should use the same task key format as `STARTUP_TASKS`:

```txt
search-ensure-settings:manual-check
```
