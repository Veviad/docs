Use **Notifications**, **Feedback**, **Audit Log**, and instance **Activity** together. Each source answers a different question; none is a complete incident record by itself.

## Notifications

Notifications are generated by background jobs for provisioning, health, backup, recovery, update, Trial, and integration failures or important outcomes.

Triage each notification by recording:

- timestamp and severity;
- customer, instance, Trial, server, backup, restore run, or schedule ID;
- current state versus the state mentioned in the notification;
- related Activity or Audit event;
- whether the underlying condition is still present.

<Warning>
  Notification read state is global, not per employee. Marking one notification or all notifications as read changes it for everyone. Notifications are cleaned up after 90 days, so preserve incident evidence elsewhere when needed.
</Warning>

## Feedback

Feedback intake captures application submissions, category, status, description, submitter context, optional attachments, internal comments, assignment, and optional GitHub issue linkage.

Suggested workflow:

<Steps>
  <Step title="Validate the report">
    Open the feedback item, review attachments carefully, and separate observed behavior from assumptions. Do not copy exposed personal or secret data into public systems.
  </Step>
  <Step title="Set ownership and status">
    Move `new` to `in_review` when someone accepts the work. Use internal comments for concise investigation context.
  </Step>
  <Step title="Link a GitHub issue when engineering work is required">
    Confirm the configured GitHub token and repository scope, sanitize the content, create the issue, and verify its number and URL.
  </Step>
  <Step title="Synchronize and close">
    Refresh the linked GitHub issue state as needed. Set feedback to `resolved`, `rejected`, or `archived` with the reason documented.
  </Step>
</Steps>

Deleting feedback removes an operational record and may affect attachment history. Prefer a terminal status unless deletion is required by an approved data-handling request.

## Instance Activity

The instance detail page shows recent events; the full Activity page provides the longer instance-specific sequence. It is the primary evidence for actions such as create, deploy, health change, backup, restore, restart, update, decommission, and failure.

When escalating, include:

- instance ID, name, domain, customer, server, and provider ID;
- current lifecycle status and health status;
- deployed and requested image versions;
- latest relevant backup or restore run IDs;
- event timestamps and sanitized error codes/messages;
- actions already attempted.

## Global Audit Log limits

The global Audit Log focuses mainly on instance lifecycle events. Writes are best effort and the list loads at most 500 entries. Settings, customer, template, email-template, and other configuration changes are not guaranteed to be fully represented.

<Note>
  Do not describe the Admin Panel audit view as a complete compliance audit. For high-impact changes, combine its evidence with Directus activity/revisions, provider logs, GitHub history, deployment records, and the incident or change ticket.
</Note>

## Close an incident only after

- the final lifecycle state and health are stable;
- customer-facing behavior is verified;
- backup or recovery consequences are understood;
- notifications no longer indicate an active problem;
- the root cause and any retry or override are documented;
- follow-up engineering or access work has a named owner.
